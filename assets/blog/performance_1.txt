Inner Inserts
=============

For this first performance analysis, I'll talk about inner mutations - specifically inserting data into the middle of a file. 
Later I'll talk about inner removals.

Definition: Inner Insert 
------------------------
  Say I have a file where each byte is just it's location on disk. Byte 0 has value 0, byte 1 has value 1:
  
  [ 0 | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 ]

  Normal files provide first class support for *inner writes*. So let's say I want to write from index 1 to 3
  with new data [ 9 | 10 | 11 ]. After a traditional "write" system call, that would look like this:

  [ 0 | 9 | 10 | 11 | 4 | 5 | 6 | 7 | 8 ]

  Notice how 1, 2 and 3 were overwritten by 9, 10, and 11

  This is what I'll call an "inner write"

  What if I want to insert data at index 1. This is actually a really common use case. 
  So now I want to insert the same data ([ 9 | 10 | 11 ]) into position 1 of the file like this:

  [ 0 | 9 | 10 | 11 | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 ]

  This is what I will call an "inner insert" (as opposed to an "inner write")


The Problem Space
-----------------

For the sake of measuring performance, an inner insert has the following parameters 

* fsize: Original File Size          - The size of the file before we make an insert 
* isize: Desired Data Insert Length  - The size of the data we want to insert into the file
* offst: Desired Data Offset         - The location in the original file we want to insert the data into 

I'll define a few other things to help our problem:

* tail:                              - The remainder of the file from file[offst ... fsize]

Solving it Naively
------------------

Let's try to solve this without smart files. In general the pattern looks like this:

1. Read in the "tail" of the file in memory - effectively breaking our file into two segments - before and after our offset
2. Write our data to the end of this truncated file 
3. Write the tail back to the end of this file. 

Let's visualize that

file = [ 0 | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 ]

1. Read in the tail 
  file = [ 0 ]
  tail = [ 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 ]

2. Write our data to the end of this truncated file 
  file = [ 0 | 9 | 10 | 11 ]
  tail = [ 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 ]

3. Write the tail back to the end of this file. 
  file = [ 0 | 9 | 10 | 11 | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 ]

>>> [FOLDED CODE WITH LABEL] Here's an unbuffered implementation that a naive programmer could write:
```
{
    i_pread_all_expect (&fp, tail, tail_size, ofst, &e);
    i_pwrite_all (&fp, tail, tail_size, ofst + insize, &e);
    i_pwrite_all (&fp, insert, insize, ofst, &e);
    i_fsync (&fp, &e);
}
```

Note that we have unbounded memory overhead in this method. The file can be any size (it's unbounded) and no matter what,
we need to "touch" all the data within the tail of the file in some way or another.

To solve this - let's try buffering. 

Buffering is what `FILE* fp = fopen(...)` gives over just Posix `int fd = open(...)`. 
Buffering is nice because it limits memory overhead by maintaining a fixed size buffer in memory

If we have a buffer of size 4096 bytes, we never need to hold more than 4096 bytes in memory at once
However, in this method, we still "touch" all the right bytes of the file. As our file grows unbounded, 
we always need to touch the right half of our file some how unless we have an os built in primitive. 
That's not ideal.

>>> [FOLDED CODE WITH LABEL] Here's a buffered implementation that a smarter programmer could write:
```
{
  {
    u64 remaining = fsize - ofst;
    u64 read_pos  = fsize;

    // Read in chunks
    while (remaining > 0)
    {
      const u64 chunk = remaining < CHUNK_SIZE ? remaining : CHUNK_SIZE;
      read_pos -= chunk;

      i_pread_all_expect (&fp, buffer, chunk, read_pos, &e);
      i_pwrite_all (&fp, buffer, chunk, read_pos + insize, &e);

      remaining -= chunk;
    }
  }
  i_pwrite_all (&fp, insert, insize, ofst, &e);
  i_fsync (&fp, &e);
}
```

Preamble: Why Numstore is more than just performant
---------------------------------------------------

Before I talk about performance, I want to mention that performance isn't the only picture,
but Numstore still dwarfs a normal system call.

The abstract:

> Numstore is a lot faster at inner mutations than anything else a developer can code up 

Fails to capture a few things that numstore does on top of being fast. It's more like:

> Numstore is a lot faster at inner mutations than anything else a developer can code up  
> AND
> 1. Numstore is ACID - unlike files
>    - you can never write 10 bytes if you requested to write 100 bytes 
>    - If you pull your plug on your computer while writing data, numstore reboots and 
>      stays in a consistent state
> 
> 2. Numstore has no extra memory overhead 
>    - You never need to care about the tail of your file when doing an inner insert
>
> 3. Numstore uses less code, refer back to our naive implementations, now refer to numstore: `smfile_insert (file, insert, ofst, insize);`
     - That's because inner inserts are first class in numstore, nothing more.

Points 1-3 are _really big_ and I don't want to gloss over them, but this post is all about performance, so lets talk speed.
